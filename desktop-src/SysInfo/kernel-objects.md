# <a name="kernel-objects"></a>Objetos de kernel

Los identificadores de objetos de kernel son específicos del proceso. Es decir, un proceso debe crear el objeto o abrir un objeto existente para obtener un identificador de objeto de kernel. El límite por proceso de los identificadores de kernel es 2 ^ 24. Sin embargo, los identificadores se almacenan en el grupo paginado, por lo que el número real de identificadores que se pueden crear se basa en la memoria disponible. El número de identificadores que se pueden crear en Windows de 32 bits es significativamente inferior a 2 ^ 24.

Cualquier proceso puede crear un nuevo identificador para un objeto de kernel existente (incluso uno creado por otro proceso), siempre que el proceso conozca el nombre del objeto y tenga acceso de seguridad al objeto. Los identificadores de objetos de kernel incluyen derechos de acceso que indican las acciones que se pueden conceder o denegar a un proceso. Una aplicación especifica derechos de acceso cuando crea un objeto u obtiene un identificador de objeto existente. Cada tipo de objeto de kernel es compatible con su propio conjunto de derechos de acceso. Por ejemplo, los identificadores de eventos pueden tener acceso set o Wait (o ambos), los identificadores de archivo pueden tener acceso de lectura o escritura (o ambos), etc. Para obtener más información, vea [objetos protegibles](/windows/desktop/SecAuthZ/securable-objects).

En la ilustración siguiente, una aplicación crea un objeto de evento. La función [**CreateEvent**](/windows/desktop/api/synchapi/nf-synchapi-createeventa) crea el objeto de evento y devuelve un identificador de objeto.

![aplicación que crea un objeto de evento](images/cshob-03.png)

Una vez creado el objeto de evento, la aplicación puede usar el identificador de evento para establecer o esperar en el evento. El identificador sigue siendo válido hasta que la aplicación cierre el identificador o termine.

La mayoría de los objetos de kernel admiten varios identificadores para un único objeto. Por ejemplo, la aplicación de la ilustración anterior podría obtener identificadores de objetos de eventos adicionales mediante la función [**OpenEvent**](/windows/desktop/api/synchapi/nf-synchapi-openeventa) , tal como se muestra en la siguiente ilustración.

![aplicación que crea un objeto de evento con varios identificadores](images/cshob-04.png)

Este método permite que una aplicación tenga identificadores con derechos de acceso diferentes. Por ejemplo, el identificador 1 podría haber establecido y esperar el acceso al evento, y el controlador 2 podría tener solo acceso de espera.

Si otro proceso conoce el nombre del evento y tiene acceso de seguridad al objeto, puede crear su propio identificador de objeto de evento mediante [**OpenEvent**](/windows/desktop/api/synchapi/nf-synchapi-openeventa). La aplicación de creación también podría duplicar uno de sus identificadores en el mismo proceso o en otro proceso mediante la función [**DuplicateHandle**](/windows/win32/api/handleapi/nf-handleapi-duplicatehandle) .

Un objeto permanece en la memoria siempre que exista al menos un identificador de objeto. En la siguiente ilustración, las aplicaciones utilizan la función [**CloseHandle**](/windows/win32/api/handleapi/nf-handleapi-closehandle) para cerrar los identificadores de objeto de evento. Cuando no hay ningún controlador de eventos, el sistema quita el objeto de la memoria, tal y como se muestra en la siguiente ilustración.

![identificadores de objeto de evento de cierre de la aplicación para quitar el objeto de la memoria](images/cshob-08.png)

El sistema administra objetos de archivo de forma ligeramente diferente a otros objetos de kernel. Los objetos de archivo contienen el puntero de archivo: el puntero al siguiente byte que se va a leer o escribir en un archivo. Cada vez que una aplicación crea un nuevo identificador de archivo, el sistema crea un nuevo objeto de archivo. Por lo tanto, más de un objeto de archivo puede hacer referencia a un único archivo en el disco, tal como se muestra en la siguiente ilustración.

![varios objetos de archivo que hacen referencia a un archivo en disco](images/cshob-09.png)

Solo a través de la duplicación o la herencia puede haber más de un identificador de archivo que haga referencia al mismo objeto de archivo, tal y como se muestra en la siguiente ilustración.

![dos identificadores de archivo hacen referencia al mismo objeto de archivo](images/cshob-10.png)

Cuando una aplicación cierra el último identificador de un objeto de kernel, el sistema quita el objeto de la memoria.
