# <a name="closing-a-file-mapping-object"></a><span data-ttu-id="bcfde-103">Cerrar un objeto de asignación de archivos</span><span class="sxs-lookup"><span data-stu-id="bcfde-103">Closing a File Mapping Object</span></span>

<span data-ttu-id="bcfde-104">Cuando un proceso ha terminado con el objeto de asignación de archivos, debe destruir todas las vistas de archivo en su espacio de direcciones mediante la función [**UnmapViewOfFile**](/windows/win32/api/memoryapi/nf-memoryapi-unmapviewoffile) para cada vista de archivo.</span><span class="sxs-lookup"><span data-stu-id="bcfde-104">When a process has finished with the file mapping object, it should destroy all file views in its address space by using the [**UnmapViewOfFile**](/windows/win32/api/memoryapi/nf-memoryapi-unmapviewoffile) function for each file view.</span></span>

<span data-ttu-id="bcfde-105">Al desasignar una vista asignada de un archivo, se invalida el intervalo ocupado por la vista en el espacio de direcciones del proceso y hace que el intervalo esté disponible para otras asignaciones.</span><span class="sxs-lookup"><span data-stu-id="bcfde-105">Unmapping a mapped view of a file invalidates the range occupied by the view in the address space of the process and makes the range available for other allocations.</span></span> <span data-ttu-id="bcfde-106">Quita la entrada del espacio de trabajo de cada página virtual sin asignar que formaba parte del espacio de trabajo del proceso y reduce el tamaño del espacio de trabajo del proceso.</span><span class="sxs-lookup"><span data-stu-id="bcfde-106">It removes the working set entry for each unmapped virtual page that was part of the working set of the process and reduces the working set size of the process.</span></span> <span data-ttu-id="bcfde-107">También disminuye el recuento de recursos compartidos de la página física correspondiente.</span><span class="sxs-lookup"><span data-stu-id="bcfde-107">It also decrements the share count of the corresponding physical page.</span></span>

<span data-ttu-id="bcfde-108">Las páginas modificadas en la vista sin asignar no se escriben en el disco hasta que su recuento de recursos compartidos llega a cero, o en otras palabras, hasta que no se asignan o se recortan de los conjuntos de trabajo de todos los procesos que comparten las páginas.</span><span class="sxs-lookup"><span data-stu-id="bcfde-108">Modified pages in the unmapped view are not written to disk until their share count reaches zero, or in other words, until they are unmapped or trimmed from the working sets of all processes that share the pages.</span></span> <span data-ttu-id="bcfde-109">Incluso después, las páginas modificadas se escriben "de forma diferida" en el disco; es decir, es posible que las modificaciones se almacenen en la memoria caché y se escriban en el disco más adelante.</span><span class="sxs-lookup"><span data-stu-id="bcfde-109">Even then, the modified pages are written "lazily" to disk; that is, modifications may be cached in memory and written to disk at a later time.</span></span> <span data-ttu-id="bcfde-110">Para minimizar el riesgo de pérdida de datos en caso de que se produzca un error de alimentación o un bloqueo del sistema, las aplicaciones deben vaciar explícitamente las páginas modificadas mediante la función [**FlushViewOfFile**](/windows/win32/api/memoryapi/nf-memoryapi-flushviewoffile) .</span><span class="sxs-lookup"><span data-stu-id="bcfde-110">To minimize the risk of data loss in the event of a power failure or a system crash, applications should explicitly flush modified pages using the [**FlushViewOfFile**](/windows/win32/api/memoryapi/nf-memoryapi-flushviewoffile) function.</span></span>

<span data-ttu-id="bcfde-111">Cuando cada proceso termina de usar el objeto de asignación de archivos y ha desasignado todas las vistas, debe cerrar el identificador del objeto de asignación de archivos y el archivo en el disco llamando a [**CloseHandle**](/windows/win32/api/handleapi/nf-handleapi-closehandle).</span><span class="sxs-lookup"><span data-stu-id="bcfde-111">When each process finishes using the file mapping object and has unmapped all views, it must close the file mapping object's handle and the file on disk by calling [**CloseHandle**](/windows/win32/api/handleapi/nf-handleapi-closehandle).</span></span> <span data-ttu-id="bcfde-112">Estas llamadas a **CloseHandle** se realizan correctamente aunque haya vistas de archivo que todavía estén abiertas.</span><span class="sxs-lookup"><span data-stu-id="bcfde-112">These calls to **CloseHandle** succeed even when there are file views that are still open.</span></span> <span data-ttu-id="bcfde-113">Sin embargo, si se dejan las vistas de archivo asignadas, se producen fugas de memoria.</span><span class="sxs-lookup"><span data-stu-id="bcfde-113">However, leaving file views mapped causes memory leaks.</span></span>
