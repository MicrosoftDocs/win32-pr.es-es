# <a name="graphics-low-level-client-support"></a>Compatibilidad con clientes Low-Level de gráficos

\[Estas funciones están sujetas a cambios en cada revisión del sistema operativo. En su lugar, use Microsoft DirectDraw y Microsoft Direct3DAPIs; estas API aíslan las aplicaciones de estos cambios del sistema operativo y ocultan muchas otras dificultades para interactuar directamente con los controladores de pantalla. Para obtener más información, consulte [Introduction to display](/windows-hardware/drivers/display/introduction-to-display--windows-2000-model-).\]

DirectDraw y Direct3D usan algunas rutinas de modo kernel para comunicarse con el sistema operativo y el controlador de pantalla.

El kernel del sistema operativo parece ser un controlador de pantalla de DirectDraw/Direct3D desde el punto de vista de los componentes de modo de usuario. Sin embargo, existen algunas diferencias entre el kernel y los controladores de pantalla. Los componentes de modo kernel conservan una copia de las estructuras que representan objetos DirectDraw/Direct3D, como dispositivos y superficies. El componente de modo de usuario de DirectDraw/Direct3D solicita la creación de esas estructuras, a las que se hace referencia con los identificadores devueltos por el modo kernel. Por consiguiente, las diferencias son principalmente en los argumentos que se pasan a las rutinas: kernel acepta identificadores, mientras que los controladores normalmente aceptan punteros.

## <a name="evolution-of-kernel-mode-entry-points"></a>Evolución de los puntos de entrada del modo kernel

DirectDraw realiza todas sus conexiones al modo kernel a través del Gdi32.dll mediante varios puntos de entrada. Estos puntos de entrada suelen simplemente serializar parámetros. Como parte de la creación de dispositivos, el Gdi32.dll devuelve punteros a rutinas dentro de sí mismo que DirectDraw y Direct3D 7,0 usan.

Direct3D 8. *x* realiza un salto limpio desde Gdi32.dll y llama directamente a los puntos de entrada en modo kernel.

## <a name="calling-the-kernel"></a>Llamar al kernel

Estos puntos de entrada son similares y, en muchos casos, son idénticos a los puntos de entrada en el propio controlador de pantalla, por lo que una comprensión de los materiales del kit de desarrollo de controladores (DDK) para DirectDraw y Direct3D es un requisito previo indispensable para usar estas funciones.

La llamada a estas funciones difiere entre los sistemas operativos. En Windows 2000, W2KUMODE. La biblioteca LIB contiene rutinas que permiten al subproceso que realiza la llamada realizar la transición al modo kernel. Estas rutinas no realizan el cálculo de referencias de parámetros y son el mecanismo de llamada ideal en Windows 2000. El mecanismo que hace que la transición al modo kernel se basa en el orden de una tabla. Esta ordenación cambiará en las revisiones del sistema operativo. Por lo tanto, no es aconsejable compilar una aplicación que se base en W2KUMODE. LIB para Windows 2000 y esperar que se ejecute sin modificar en Windows XP. En la sección siguiente se describe un mecanismo para crear la portabilidad del sistema operativo.

En Windows XP, D3D8THK.DLL exporta todas las funciones de modo kernel, pero con un nombre ligeramente representativo. En el ejemplo siguiente se muestra cómo las aplicaciones pueden asignar alias a las funciones de cliente de bajo nivel mediante el preprocesador. Antes de hacerlo, vincule dinámicamente a D3D8THK.DLL.


```

#define NtGdiDdUnlock OsThunkDdUnlock
#include "ntgdi.h"      // Brings in the definition of "NtGdiDdUnlock", 
.                       //   which is now seen as OsThunkDdUnlock.
.
.
OsThunkDdUnlock(hSurface, puUnlockData);
```



## <a name="writing-portable-applications"></a>Escribir aplicaciones portátiles

Hay varias técnicas disponibles para convertir una aplicación en portable en los distintos sistemas operativos. La más sólida es instalar el paquete redistribuible de Microsoft DirectX 8. x en Windows 2000 como parte del proceso de instalación de su propia aplicación. Esto garantiza que D3D8THK.DLL estará disponible en los sistemas 2000 de Windows y que permita una única ruta de acceso de código en la aplicación, ya que puede usar la misma técnica que se describe en Windows XP anterior.

Las aplicaciones también pueden crear su propio archivo DLL de aislamiento e implementar dos versiones de ese archivo DLL, una para Windows 2000 con la técnica mostrada anteriormente y otra para Windows XP que usa D3D8THK.DLL.

## <a name="device-creation"></a>Creación de dispositivos

DirectDraw y Direct3D primero deben crear una instancia de la abstracción del kernel del objeto de controlador. En Gdi32.dll y Ddraw.dll, esto se logra mediante una llamada a [**DdCreateDirectDrawObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddcreatedirectdrawobject), [**DdQueryDirectDrawObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddquerydirectdrawobject) y [**DdReenableDirectDrawObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddreenabledirectdrawobject). En Direct3D 8. *x*, esto se logra mediante una llamada a [**NtGdiDdCreateDirectDrawObject**](-dxgkernel-ntgdiddcreatedirectdrawobject.md), [**NtGdiDdQueryDirectDrawObject**](-dxgkernel-ntgdiddquerydirectdrawobject.md) y [**NtGdiDdReenableDirectDrawObject**](-dxgkernel-ntgdiddreenabledirectdrawobject.md). Tenga en cuenta que los puntos de entrada de GdiEntry \* son simplemente cálculos de referencia para los puntos de entrada en modo kernel.

Después de llamar a estas funciones, se devuelve un identificador que representa la abstracción del objeto de kernel de un dispositivo DirectDraw. Este objeto de kernel representa una instancia de controlador propiedad del controlador de pantalla. Este identificador se puede utilizar en llamadas posteriores, como [**DdCreateSurfaceObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddcreatesurfaceobject) (o [**NtGdiDdCreateSurfaceObject**](-dxgkernel-ntgdiddcreatesurfaceobject.md) en Direct3D 8.*x*) para compilar más objetos.

## <a name="kernel-objects"></a>Objetos de kernel

Algunos puntos de entrada se utilizan para administrar las representaciones del modo kernel de ciertos objetos. [**DdCreateSurfaceObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddcreatesurfaceobject) (o [**NtGdiDdCreateSurfaceObject**](-dxgkernel-ntgdiddcreatesurfaceobject.md) en Direct3D 8.*x*) es un buen ejemplo. Este punto de entrada crea un objeto de modo kernel que representa una parte de la memoria del sistema. El punto de entrada devuelve un identificador y este identificador se puede usar en las llamadas de dibujo posteriores. El kernel deriva un puntero a su propia estructura de estos identificadores y pasa ese puntero al controlador de pantalla, que, a continuación, lleva a cabo la operación.
