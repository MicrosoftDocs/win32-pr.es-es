# <a name="graphics-low-level-client-support"></a>Compatibilidad con Low-Level gráficos

\[Estas funciones están sujetas a cambios con cada revisión del sistema operativo. En su lugar, use Microsoft DirectDraw y Microsoft Direct3DAPIs; estas API aíslan las aplicaciones de estos cambios del sistema operativo y ocultan muchas otras dificultades implicadas en la interacción directa con los controladores de pantalla. Para obtener más información, vea [Introduction to Display](/windows-hardware/drivers/display/introduction-to-display--windows-2000-model-).\]

DirectDraw y Direct3D usan algunas rutinas en modo kernel para comunicarse con el sistema operativo y el controlador de pantalla.

El kernel del sistema operativo parece ser un controlador de visualización DirectDraw/Direct3D desde el punto de vista de los componentes en modo de usuario. Sin embargo, existen algunas diferencias entre los controladores de kernel y de visualización. Los componentes del modo kernel mantienen una copia de estructuras que representan objetos DirectDraw/Direct3D, como dispositivos y superficies. El componente de modo de usuario de DirectDraw/Direct3D solicita la creación de esas estructuras, a las que se hace referencia con los identificadores devueltos por el modo kernel. Por lo tanto, las diferencias se encuentran principalmente en los argumentos pasados a las rutinas: el kernel acepta identificadores, mientras que los controladores suelen aceptar punteros.

## <a name="evolution-of-kernel-mode-entry-points"></a>Evolución de los puntos de entrada del modo kernel

DirectDraw realiza todas sus conexiones al modo kernel a través del Gdi32.dll mediante varios puntos de entrada. Estos puntos de entrada normalmente solo serializan los parámetros. Como parte de la creación de dispositivos, el Gdi32.dll devuelve punteros a rutinas dentro de sí que directdraw y Direct3D 7.0 usan.

Direct3D 8. *x* realiza una interrupción limpia de Gdi32.dll y llama directamente a los puntos de entrada en modo kernel.

## <a name="calling-the-kernel"></a>Llamada al kernel

Estos puntos de entrada son similares y, en muchos casos, idénticos a los puntos de entrada del propio controlador de pantalla, por lo que un conocimiento de los materiales del Kit de desarrollo de controladores (DDK) para DirectDraw y Direct3D es un requisito previo esencial para usar estas funciones.

Llamar a estas funciones difiere entre los sistemas operativos. En Windows 2000, W2KUMODE. La biblioteca LIB contiene rutinas que permiten al subproceso que realiza la llamada realizar la transición al modo kernel. Estas rutinas no hacen ninguna referencia de parámetros y son el mecanismo de llamada ideal en Windows 2000. El mecanismo que realiza la transición al modo kernel se basa en el orden de una tabla. Esta ordenación cambiará en todas las revisiones del sistema operativo. Por lo tanto, no es aconsejable compilar una aplicación que se base en W2KUMODE. LIB para Windows 2000 y espera que se ejecute sin modificar en Windows XP. En la sección siguiente se describe un mecanismo para crear la portabilidad del sistema operativo.

En Windows XP, D3D8THK.DLL exporta todas las funciones del modo kernel, pero con un nombre ligeramente decorado. En el ejemplo siguiente se muestra cómo las aplicaciones pueden crear alias para las funciones cliente de bajo nivel mediante el preprocesador. Antes de hacerlo, vincule dinámicamente a D3D8THK.DLL.


```

#define NtGdiDdUnlock OsThunkDdUnlock
#include "ntgdi.h"      // Brings in the definition of "NtGdiDdUnlock", 
.                       //   which is now seen as OsThunkDdUnlock.
.
.
OsThunkDdUnlock(hSurface, puUnlockData);
```



## <a name="writing-portable-applications"></a>Escritura de aplicaciones portátiles

Hay varias técnicas disponibles para que una aplicación sea portátil en todos los sistemas operativos. Lo más sólido es instalar el paquete redistribuible de Microsoft DirectX 8.x en Windows 2000 como parte del proceso de instalación de su propia aplicación. Esto garantiza que D3D8THK.DLL estará disponible en sistemas de Windows 2000 y permite una ruta de acceso de código única en la aplicación, ya que puede usar la misma técnica que se describió para Windows XP anterior.

Las aplicaciones también pueden crear su propio archivo DLL aislado e implementar dos versiones de ese archivo DLL, una para Windows 2000 mediante la técnica mostrada anteriormente y otra para Windows XP que usa D3D8THK.DLL.

## <a name="device-creation"></a>Creación de dispositivos

DirectDraw y Direct3D deben crear primero una instancia de la abstracción del kernel del objeto de controlador. En Gdi32.dll y Ddraw.dll, esto se logra mediante una llamada a [**DdCreateDirectDrawObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddcreatedirectdrawobject), [**DdQueryDirectDrawObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddquerydirectdrawobject) y [**DdReenableDirectDrawObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddreenabledirectdrawobject). En Direct3D 8. *x*, esto se logra mediante una llamada [**a NtGdiDdCreateDirectDrawObject**](-dxgkernel-ntgdiddcreatedirectdrawobject.md), [**NtGdiDdQueryDirectDrawObject**](-dxgkernel-ntgdiddquerydirectdrawobject.md) y [**NtGdiDdReenableDirectDrawObject**](-dxgkernel-ntgdiddreenabledirectdrawobject.md). Tenga en cuenta que los puntos de entrada de GdiEntry \* son simplemente marshallers para los puntos de entrada en modo kernel.

Después de llamar a estas funciones, se devuelve un identificador que representa la abstracción del objeto kernel de un dispositivo DirectDraw. Este objeto de kernel representa una instancia de controlador propiedad del controlador para mostrar. Este identificador se puede usar en llamadas posteriores como [**DdCreateSurfaceObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddcreatesurfaceobject) (o [**NtGdiDdCreateSurfaceObject**](-dxgkernel-ntgdiddcreatesurfaceobject.md) en Direct3D 8.*x*) para compilar más objetos.

## <a name="kernel-objects"></a>Objetos kernel

Se usan algunos puntos de entrada para administrar las representaciones en modo kernel de determinados objetos. [**DdCreateSurfaceObject**](/windows/desktop/api/Ddrawgdi/nf-ddrawgdi-ddcreatesurfaceobject) (o [**NtGdiDdCreateSurfaceObject**](-dxgkernel-ntgdiddcreatesurfaceobject.md) en Direct3D 8.*x*) es un buen ejemplo. Este punto de entrada crea un objeto en modo kernel que representa un fragmento de memoria del sistema. El punto de entrada devuelve un identificador y este identificador se puede usar en llamadas de dibujo posteriores. El kernel deriva un puntero a su propia estructura a partir de estos identificadores y pasa ese puntero al controlador para mostrar, que luego lleva a cabo la operación.
