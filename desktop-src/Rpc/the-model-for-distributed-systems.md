# <a name="the-model-for-distributed-systems"></a><span data-ttu-id="032a7-103">El modelo para sistemas distribuidos</span><span class="sxs-lookup"><span data-stu-id="032a7-103">The Model for Distributed Systems</span></span>

<span data-ttu-id="032a7-104">Tradicionalmente, tener un sistema monolítico en varios equipos significaba dividir el sistema en componentes independientes de cliente y servidor.</span><span class="sxs-lookup"><span data-stu-id="032a7-104">Traditionally, having a monolithic system run across multiple computers meant splitting the system into separate client and server components.</span></span> <span data-ttu-id="032a7-105">En estos sistemas, el componente de cliente administraba la interfaz de usuario y el servidor proporcionaba el procesamiento de back-end, como el acceso a la base de datos, la impresión, etc.</span><span class="sxs-lookup"><span data-stu-id="032a7-105">In such systems, the client component handled the user interface and the server provided back-end processing, such as database access, printing, and so on.</span></span> <span data-ttu-id="032a7-106">A medida que los equipos proliferan, descartaban el costo y se conectaban a través de redes de ancho de banda cada vez mayores, la división de los sistemas de software en varios componentes resultaba más cómoda, con cada componente que se ejecutaba en un equipo diferente y realizaba una función especializada.</span><span class="sxs-lookup"><span data-stu-id="032a7-106">As computers proliferated, dropped in cost, and became connected by ever-higher bandwidth networks, splitting software systems into multiple components became more convenient, with each component running on a different computer and performing a specialized function.</span></span> <span data-ttu-id="032a7-107">Este enfoque simplifica el desarrollo, la administración, la administración y, a menudo, el rendimiento y la solidez mejorados, ya que los errores en un equipo no deshabilitaron necesariamente todo el sistema.</span><span class="sxs-lookup"><span data-stu-id="032a7-107">This approach simplified development, management, administration, and often improved performance and robustness, since failure in one computer did not necessarily disable the entire system.</span></span>

<span data-ttu-id="032a7-108">En muchos casos, el sistema aparece en el cliente como una nube opaca que realiza las operaciones necesarias, aunque el sistema distribuido se compone de nodos individuales, tal y como se muestra en la ilustración siguiente.</span><span class="sxs-lookup"><span data-stu-id="032a7-108">In many cases the system appears to the client as an opaque cloud that performs the necessary operations, even though the distributed system is composed of individual nodes, as illustrated in the following figure.</span></span>

![los clientes obtienen acceso a los servicios de un sistema de servidores RPC que aparece como una nube opaca para clientes externos](images/indy-nodes.png)

<span data-ttu-id="032a7-110">La opacidad de la nube se mantiene porque las operaciones de cálculo se invocan en nombre del cliente.</span><span class="sxs-lookup"><span data-stu-id="032a7-110">The opacity of the cloud is maintained because computing operations are invoked on behalf of the client.</span></span> <span data-ttu-id="032a7-111">Como tal, los clientes pueden localizar un equipo (un *nodo*) dentro de la nube y solicitar una operación determinada; al realizar la operación, ese equipo puede invocar la funcionalidad en otros equipos dentro de la nube sin exponer los pasos adicionales, o el equipo en el que se realizaron, al cliente.</span><span class="sxs-lookup"><span data-stu-id="032a7-111">As such, clients can locate a computer (a *node*) within the cloud and request a given operation; in performing the operation, that computer can invoke functionality on other computers within the cloud without exposing the additional steps, or the computer on which they were carried out, to the client.</span></span>

<span data-ttu-id="032a7-112">Con este paradigma, la mecánica de un sistema distribuido, similar a la nube, se puede dividir en muchos intercambios de paquetes individuales o en conversaciones entre nodos individuales.</span><span class="sxs-lookup"><span data-stu-id="032a7-112">With this paradigm, the mechanics of a distributed, cloud-like system can be broken down into many individual packet exchanges, or conversations between individual nodes.</span></span>

<span data-ttu-id="032a7-113">Los sistemas cliente-servidor tradicionales tienen dos nodos con roles y responsabilidades fijos.</span><span class="sxs-lookup"><span data-stu-id="032a7-113">Traditional client-server systems have two nodes with fixed roles and responsibilities.</span></span> <span data-ttu-id="032a7-114">Los sistemas distribuidos modernos pueden tener más de dos nodos y sus roles suelen ser dinámicos.</span><span class="sxs-lookup"><span data-stu-id="032a7-114">Modern-distributed systems can have more than two nodes, and their roles are often dynamic.</span></span> <span data-ttu-id="032a7-115">En una conversación, un nodo puede ser un cliente, mientras que en otra conversación el nodo puede ser el servidor.</span><span class="sxs-lookup"><span data-stu-id="032a7-115">In one conversation a node can be a client, while in another conversation the node can be the server.</span></span> <span data-ttu-id="032a7-116">En muchos casos, el consumidor más avanzado de la funcionalidad expuesta es un cliente con un usuario que se encuentra en un teclado y ve la salida.</span><span class="sxs-lookup"><span data-stu-id="032a7-116">In many cases, the ultimate consumer of the exposed functionality is a client with a user sitting at a keyboard, watching the output.</span></span> <span data-ttu-id="032a7-117">En otros casos, las funciones del sistema distribuido están desatendidas y realizan operaciones en segundo plano.</span><span class="sxs-lookup"><span data-stu-id="032a7-117">In other cases the distributed system functions unattended, performing background operations.</span></span>

<span data-ttu-id="032a7-118">Es posible que el sistema distribuido no tenga clientes y servidores dedicados para cada intercambio de paquetes determinado, pero es importante recordar que hay un llamador (o iniciador, cualquiera de los cuales a menudo se conoce como cliente).</span><span class="sxs-lookup"><span data-stu-id="032a7-118">The distributed system may not have dedicated clients and servers for each particular packet exchange, but it is important to remember there is a caller, (or initiator, either of which is often referred to as the client).</span></span> <span data-ttu-id="032a7-119">También hay el destinatario de la llamada (a menudo denominado servidor).</span><span class="sxs-lookup"><span data-stu-id="032a7-119">There is also the recipient of the call (often referred to as the server).</span></span> <span data-ttu-id="032a7-120">No es necesario tener intercambios de paquetes bidireccionales en el formato de solicitud-respuesta de un sistema distribuido; a menudo, los mensajes solo se envían de una manera.</span><span class="sxs-lookup"><span data-stu-id="032a7-120">It is not necessary to have two-way packet exchanges in the request-reply format of a distributed system; often messages are sent only one way.</span></span>
