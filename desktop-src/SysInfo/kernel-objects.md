# <a name="kernel-objects"></a>Objetos kernel

Los identificadores de objetos kernel son específicos del proceso. Es decir, un proceso debe crear el objeto o abrir un objeto existente para obtener un identificador de objeto de kernel. El límite por proceso en los identificadores de kernel es 2^24. Sin embargo, los identificadores se almacenan en el grupo paginado, por lo que el número real de identificadores que puede crear se basa en la memoria disponible. El número de identificadores que se pueden crear en un Windows de 32 bits es significativamente menor que 2^24.

Cualquier proceso puede crear un nuevo identificador para un objeto de kernel existente (incluso uno creado por otro proceso), siempre que el proceso conozca el nombre del objeto y tenga acceso de seguridad al objeto. Los identificadores de objetos kernel incluyen derechos de acceso que indican las acciones que se pueden conceder o denegar a un proceso. Una aplicación especifica los derechos de acceso cuando crea un objeto u obtiene un identificador de objeto existente. Cada tipo de objeto de kernel admite su propio conjunto de derechos de acceso. Por ejemplo, los identificadores de eventos pueden tener acceso establecido o de espera (o ambos), los identificadores de archivo pueden tener acceso de lectura o escritura (o ambos), y así sucesivamente. Para obtener más información, [vea Objetos protegibles](/windows/desktop/SecAuthZ/securable-objects).

En la ilustración siguiente, una aplicación crea un objeto de evento. La [**función CreateEvent**](/windows/desktop/api/synchapi/nf-synchapi-createeventa) crea el objeto de evento y devuelve un identificador de objeto.

![aplicación que crea un objeto de evento](images/cshob-03.png)

Una vez creado el objeto de evento, la aplicación puede usar el identificador de evento para establecer o esperar en el evento. El identificador sigue siendo válido hasta que la aplicación cierra el identificador o finaliza.

La mayoría de los objetos de kernel admiten varios identificadores para un solo objeto. Por ejemplo, la aplicación de la ilustración anterior podría obtener identificadores de objetos de evento adicionales mediante la [**función OpenEvent,**](/windows/desktop/api/synchapi/nf-synchapi-openeventa) como se muestra en la ilustración siguiente.

![aplicación que crea un objeto de evento con varios identificadores](images/cshob-04.png)

Este método permite que una aplicación tenga identificadores con derechos de acceso diferentes. Por ejemplo, el identificador 1 podría haber establecido y esperar el acceso al evento, y el identificador 2 podría tener solo acceso de espera.

Si otro proceso conoce el nombre del evento y tiene acceso de seguridad al objeto , puede crear su propio identificador de objeto de evento mediante [**OpenEvent**](/windows/desktop/api/synchapi/nf-synchapi-openeventa). La aplicación de creación también podría duplicar uno de sus identificadores en el mismo proceso o en otro mediante la [**función DuplicateHandle.**](/windows/win32/api/handleapi/nf-handleapi-duplicatehandle)

Un objeto permanece en memoria siempre que exista al menos un identificador de objeto. En la ilustración siguiente, las aplicaciones usan la [**función CloseHandle**](/windows/win32/api/handleapi/nf-handleapi-closehandle) para cerrar sus identificadores de objeto de evento. Cuando no hay identificadores de eventos, el sistema quita el objeto de la memoria, como se muestra en la ilustración siguiente.

![identificadores de objeto de evento de cierre de aplicación para quitar el objeto de la memoria](images/cshob-08.png)

El sistema administra los objetos de archivo de forma ligeramente diferente de otros objetos de kernel. Los objetos de archivo contienen el puntero de archivo: el puntero al byte siguiente que se va a leer o escribir en un archivo. Cada vez que una aplicación crea un nuevo identificador de archivo, el sistema crea un nuevo objeto de archivo. Por lo tanto, más de un objeto de archivo puede hacer referencia a un único archivo en disco, como se muestra en la siguiente ilustración.

![varios objetos de archivo que hacen referencia a un archivo en disco](images/cshob-09.png)

Solo mediante la duplicación o la herencia puede más de un identificador de archivo hacer referencia al mismo objeto de archivo, como se muestra en la ilustración siguiente.

![Dos identificadores de archivo hacen referencia al mismo objeto de archivo](images/cshob-10.png)

Cuando una aplicación cierra el último identificador a un objeto kernel, el sistema quita el objeto de la memoria.
